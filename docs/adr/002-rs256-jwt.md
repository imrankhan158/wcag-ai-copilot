# ADR 002: RS256 Asymmetric JWT for Inter-Service Authentication

- **Status**: Accepted
- **Date**: 2026-06-11
- **Decision Makers**: Architecture Team

## Context

Migrating from a monolith to microservices requires a distributed authentication strategy. In the monolith, the auth module shared the same process and database connection pool as all other routes. In the microservices architecture, each service needs to independently verify user identity without coupling to the Auth Service on every request.

We evaluated three approaches:

| Option | Pros | Cons |
|---|---|---|
| **HS256 (Symmetric JWT)** | Simple, fast verification | Shared secret across all services — if one service is compromised, all tokens are forgeable |
| **RS256 (Asymmetric JWT)** | Only Auth Service holds private key; all others verify with public key | Slightly larger tokens (~800B vs ~300B), slower signing |
| **Opaque Tokens + Auth Service RPC** | Centralized validation, easy revocation | Auth Service becomes a single point of failure; every request adds network latency |

## Decision

**Use RS256 asymmetric JWT tokens.**

- The **Auth Service** signs tokens with an RSA private key
- The **API Gateway** and all downstream services verify tokens using the corresponding RSA public key
- The public key is distributed via environment variable (`JWT_PUBLIC_KEY`)

## Token Flow

```
┌──────────┐    POST /auth/login     ┌──────────────┐
│  Client   │ ──────────────────────▶ │ Auth Service  │
│           │ ◀────────────────────── │ (Private Key) │
│           │    { access_token }     └──────────────┘
│           │                                │
│           │    Authorization: Bearer ...    │
│           │ ──────────────────────▶ ┌──────────────┐
│           │                        │ API Gateway   │
│           │                        │ (Public Key)  │
│           │                        │ Verifies JWT  │
│           │                        │ Injects:      │
│           │                        │  X-User-ID    │
│           │                        │  X-User-Email │
│           │                        └──────┬───────┘
│           │                               │
│           │                        ┌──────▼───────┐
│           │                        │ Downstream    │
│           │                        │ Services      │
│           │                        │ Trust headers │
│           │                        └──────────────┘
└──────────┘
```

## Token Structure

```json
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "exp": 1718300000,
  "iat": 1718296400,
  "type": "access"
}
```

- **Access tokens**: 30-minute expiry
- **Refresh tokens**: 7-day expiry, stored in Redis with blacklist support for logout

## Consequences

### Positive
- **Zero-trust verification**: Downstream services verify tokens locally without network calls to Auth Service
- **Key isolation**: Private key exists only in the Auth Service; compromise of any other service cannot forge tokens
- **Scalability**: Token verification is a CPU-only operation — no database or Redis lookup required for access tokens
- **Gateway-level identity injection**: Services downstream of the gateway can trust `X-User-ID` / `X-User-Email` headers without re-parsing the JWT

### Negative
- **Token revocation delay**: Access tokens remain valid until expiry (30 min) even after logout. Mitigated by short TTL and Redis blacklist check at the Gateway level
- **Key rotation complexity**: Rotating the RSA key pair requires coordinated deployment of Auth Service + all services that hold the public key. Mitigated by distributing via environment variables managed through Kubernetes Secrets

### Mitigations
- Redis-based token blacklist at the API Gateway for immediate revocation of compromised tokens
- Rate limiter (Redis Lua script with token bucket) at the Gateway to prevent brute-force token usage
- Kubernetes `Secret` resource for public key distribution with rolling updates
